## Junior-engineer Q&A

### Q1 — Why stop at `v0.10.0` and not push for `v1.0.0`?

**A.** `v1.0.0` carries connotations of API stability + a "ready for outside contributors" promise. The user wanted to flip the repo public without committing to that promise. `v0.10.0` is the honest framing: feature-complete, polished, well-tested, but the user reserves the right to break things post-public without a major-version bump. The pre-1.0 SemVer convention (each minor = one phase) maps to this naturally.

### Q2 — Why is the CI badge a 404 right now?

**A.** GitHub Actions native badge URLs require the workflow file to be readable. While the repo is private, the badge URL returns 404 to unauthenticated users (and shields.io / GitHub render that as "no status"). The badge will work the moment the repo flips to public — no code change required. The README badge stays in place; it just renders unhelpfully until then. (PR #50 worker noted this explicitly.)

### Q3 — Why are the screenshots empty-state instead of populated?

**A.** Privacy. The user's live `postdoc.db` carries real position-tracking data. Screenshotting it and committing the PNGs publicly would leak that activity. Three options were considered: (A) empty-state captures, (B) build a fake-seed-data script first, (C) commit real data. The user chose (A) — fastest, safe, honest "what new users see on first run". (B) was estimated at +1h scope and rejected for v0.10.0; it's a reasonable post-public addition if the empty-state framing under-sells the app.

### Q4 — Why is the coverage badge static (not codecov)?

**A.** Codecov requires a per-PR upload step in CI + an account on codecov.io + a token. Static badge with manual updates is simpler for a solo project where a coverage shift big enough to update the README is rare. If the project ever sets up codecov properly, swap the badge URL — the README placement stays identical.

### Q5 — The `v0.9.0` and `v0.10.0` releases shipped on the same day. Are these patch-bump candidates instead?

**A.** No. The project's pre-1.0 convention is "each minor bump marks one completed phase or tier" (per `GUIDELINES §11`). `v0.9.0` shipped the v1.0-rc schema cleanup (a breaking-change-class migration; auto-applied) AND the publish-readiness scaffolding (README, LICENSE, repo rename) — that's a discrete completed unit. `v0.10.0` ships the *polish layer over* the publish scaffolding (badges, screenshots, SECURITY, dev-tooling matrix) — a discrete completed unit on top. Both warrant minor bumps under the convention. Patch versions would add semantic noise without adding clarity.

### Q6 — The CI matrix expansion claims to cover 3.11–3.14, but local-environment verification only ran 3.14. Are there latent compatibility issues?

**A.** Possible but very low risk. The codebase uses no Python 3.12+ syntax (no PEP 695 type parameters, no PEP 701 f-string features specific to 3.12+, no PEP 698 `@override`). Pinned dependencies (`pandas 3.0`, `numpy 2.4`, `streamlit 1.57.0`) all support 3.11+. The CI matrix expansion is the verification — when PR #51's CI runs on push, it exercises 3.11/3.12/3.13/3.14 in parallel; if any version breaks, fix-forward in a follow-up patch. The ship-this-version risk is bounded.
